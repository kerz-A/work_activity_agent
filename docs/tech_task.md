# **Агент анализа скриншотов из таск‑трекера**

Агент, который оцифровывает скриншоты, классифицирует активность, находит аномалии и готовит отчёты для менеджера, но не принимает дисциплинарных решений автоматически.

Тут можно [посмотреть](https://docs.google.com/document/d/1n-OaaHXrUvhOD5TpQUdWWZ4vIMe35jSOWNqNx8Luu78/edit?usp=sharing) частично готовые скилы

---

# **1\. Что делает агент**

Скриншоты из AppTask  
\+ задачи  
\+ сотрудник  
\+ проект  
\+ трекнутое время  
\+ активность мыши/клавиатуры  
\+ URL/приложения, если доступны  
       ↓  
OCR и vision-анализ  
       ↓  
описание происходящего  
       ↓  
классификация активности  
       ↓  
сравнение с задачей  
       ↓  
поиск аномалий  
       ↓  
risk score  
       ↓  
отчёт менеджеру  
       ↓  
ручная проверка

Главная цель:

Понять, соответствует ли зафиксированная активность заявленной рабочей задаче, и найти ситуации, которые требуют проверки менеджером.

---

# **2\. Что агент анализирует на скриншоте**

## **Базовая оцифровка**

Агент должен описывать:

\- какое приложение открыто;  
\- какой сайт открыт;  
\- какой документ / макет / код / интерфейс виден;  
\- есть ли рабочий контекст;  
\- есть ли видимая задача, проект, репозиторий, CRM, Figma, IDE, браузер;  
\- есть ли мессенджер, почта, соцсети, вакансии, личные сайты;  
\- меняется ли картинка между скриншотами;  
\- видны ли признаки работы: редактирование, код, дизайн, задачи, переписка по проекту;  
\- есть ли нерабочая активность.

## **Пример оцифровки**

{  
 "screenshot\_id": "scr\_2026\_04\_22\_10\_15\_00",  
 "employee": "developer\_1",  
 "project": "Client CRM",  
 "tracked\_task": "Fix payment retry bug",  
 "visible\_app": "VS Code",  
 "visible\_context": "TypeScript file retry.service.ts",  
 "visible\_activity": "Code editor with payment retry logic",  
 "ocr\_summary": "Function shouldRetry(error) and tests are visible",  
 "work\_relevance": "high",  
 "risk\_flags": \[\],  
 "confidence": 0.87  
}

# **3\. Классификация активности**

Агенту нужна не бинарная логика `работает / не работает`, а категории.

| Категория | Пример |
| ----- | ----- |
| Productive work | IDE, Figma, Jira, CRM, документация, рабочий браузер |
| Project communication | Slack/Telegram/почта по рабочему проекту |
| Research | документация, StackOverflow, API docs, статьи по задаче |
| Admin work | табели, отчёты, CRM, планирование |
| Neutral / unclear | экран не даёт однозначного вывода |
| Idle / static | долго не меняется экран |
| Non-work | соцсети, видео, игры, личные сайты |
| Job search signal | hh.ru, LinkedIn вакансии, резюме, собеседования |
| Other project signal | чужой репозиторий, другая CRM, фриланс-биржа |
| Sensitive / private | личная переписка, банковские данные, документы |
| Needs human review | всё, что может привести к санкциям |

---

# **4\. Что можно считать сигналами риска**

Важно: агент должен фиксировать **наблюдаемые факты**, а не делать окончательные обвинения.

Плохо:

Сотрудник фармит время.  
Сотрудник ищет новую работу.  
Сотрудник работает на другой проект.

Правильно:

На 6 из 12 скриншотов за час открыт сайт с вакансиями.  
На 8 последовательных скриншотах экран не менялся.  
На скриншотах виден репозиторий, не связанный с текущим проектом.  
Активность не соответствует задаче, требуется проверка менеджером.

## **Возможные risk flags**

| Flag | Что значит |
| ----- | ----- |
| `static_screen_long_period` | экран не меняется длительное время |
| `unrelated_website` | сайт не связан с задачей |
| `job_search_site` | открыт сайт вакансий / резюме |
| `freelance_platform` | открыта фриланс‑биржа |
| `other_project_tool` | открыт другой таск‑трекер / чужой репозиторий |
| `personal_messenger` | личная переписка, если не рабочий канал |
| `entertainment_content` | видео, игры, соцсети |
| `low_task_relevance` | активность плохо связана с заявленной задачей |
| `sensitive_data_detected` | видны данные, которые нельзя распространять |
| `manual_review_required` | нужна проверка человеком |

# **5\. Как агент должен определять «фарм времени»**

Он не должен выносить вердикт сам. Он должен считать **индекс подозрительности**.

## **Пример логики**

Time Farming Risk Score \=  
 25% доля статичных скриншотов  
\+ 20% несоответствие активности задаче  
\+ 15% доля нерабочих сайтов/приложений  
\+ 15% отсутствие признаков прогресса  
\+ 10% повторяемость паттерна по дням  
\+ 10% расхождение с трекнутым временем  
\+ 5% ручные замечания менеджера

## **Пример вывода**

| Категория | Что ищет агент |
| ----- | ----- |
| Низкая активность | долго неизменный экран, отсутствие прогресса |
| Несоответствие задаче | другая тема, другой проект, другой инструмент |
| Нерабочие сайты | развлечения, соцсети, игры |
| Поиск работы | сайты вакансий, резюме, собеседования |
| Другие проекты | фриланс‑биржи, чужие репозитории, чужие таск‑трекеры |

| Риски безопасности | токены, пароли, приватные ключи на экране |
| Потенциальная утечка | документы клиента в неподходящих каналах |

# **13\. Важный принцип: скриншот ≠ результат**

Скриншоты показывают активность, но не всегда показывают результат.

Например:

\- разработчик может думать над архитектурой без видимого движения;  
\- дизайнер может анализировать референсы;  
\- менеджер может быть на созвоне;  
\- аналитик может читать документацию;  
\- сотрудник может открыть личный сайт на 2 минуты во время перерыва.

Поэтому агент должен использовать скриншоты как **сигнал**, а не как единственный источник истины.

Правильная связка:

скриншоты  
\+ задачи  
\+ коммиты  
\+ закрытые таски  
\+ pull requests  
\+ Figma history  
\+ CRM-активность  
\+ сообщения по проекту  
\+ фактический результат

# **14\. Scoring продуктивности**

## **Пример модели**

Work Activity Score \=  
 25% соответствие задаче  
\+ 20% наличие результата  
\+ 15% активность в рабочих инструментах  
\+ 15% динамика между скриншотами  
\+ 10% проектная коммуникация  
\+ 10% регулярность работы  
\+ 5% отсутствие risk flags

## **Но**

Не нужно делать один «рейтинг сотрудника» и использовать его для наказаний. Лучше:

\- отчёт по рискам;  
\- интервалы для проверки;  
\- доказательства;  
\- confidence;  
\- комментарий менеджера;  
\- право сотрудника объяснить контекст.

# **17\. Риски такого агента**

| Риск | Что может пойти не так | Защита |
| ----- | ----- | ----- |
| Ложные обвинения | Агент ошибочно интерпретирует скрин | Human review |
| Privacy‑нарушение | Захват личной переписки или данных | Redaction \+ minimization |
| Юридический риск | Нет уведомления / регламента | Legal review |
| Недоверие команды | Люди воспринимают как тотальную слежку | Прозрачные правила |
| Ошибки модели | OCR/vision неправильно распознали сайт | Confidence score |
| Сбор лишнего | Хранится слишком много скринов | Retention policy |
| Автоматические санкции | Решение без человека | Запрет auto‑discipline |
| Смещение оценки | «Думает» \= «не работает» | Учитывать результат, а не только экран |
| Захват данных клиентов | На скринах документы и доступы | Masking \+ ограничение доступа |

# **22\. Улучшенный системный промпт агента**

Ты — Work Activity Screenshot Analysis Agent.

Твоя задача — анализировать скриншоты из таск‑трекера, оцифровывать их в текст, классифицировать активность и находить эпизоды, которые требуют ручной проверки менеджером.

Ты не выносишь дисциплинарных решений.  
Ты не обвиняешь сотрудников.  
Ты не делаешь выводы о намерениях.  
Ты фиксируешь только наблюдаемые факты и уровень уверенности.

Для каждого скриншота определи:  
1\. Время.  
2\. Сотрудника.  
3\. Проект.  
4\. Задачу.  
5\. Видимое приложение.  
6\. Видимый сайт или документ.  
7\. Краткое описание происходящего.  
8\. OCR ключевого текста.  
9\. Тип активности:  
  \- productive\_work;  
  \- project\_communication;  
  \- research;  
  \- admin\_work;  
  \- neutral\_unclear;  
  \- idle\_static;  
  \- non\_work;  
  \- sensitive\_private;  
  \- manual\_review\_required.  
10\. Соответствие задаче:  
  \- high;  
  \- medium;  
  \- low;  
  \- unclear.  
11\. Risk flags.  
12\. Confidence score.

Правила:  
\- Не читать и не пересказывать личную переписку подробнее, чем нужно для классификации.  
\- Маскировать персональные и чувствительные данные.  
\- Не использовать один скриншот для обвинения.  
\- Не писать “сотрудник фармит время”.  
\- Писать “обнаружены признаки, требующие проверки”.  
\- Не делать автоматических санкций.  
\- Всегда отделять факт, гипотезу и рекомендацию.

Формат вывода:  
\- таблица по скриншотам;  
\- timeline по сотруднику;  
\- список risk episodes;  
\- общий risk score;  
\- evidence;  
\- рекомендации менеджеру;  
\- список эпизодов для ручной проверки.

# **23\. Как правильно сформулировать агент в продукте**

**Агент анализа скриншотов из AppTask** — это агент, который:

\- просматривает скриншоты рабочего времени;  
\- оцифровывает содержимое скриншотов в текст;  
\- определяет приложения, сайты, документы и рабочий контекст;  
\- сравнивает активность с задачей и проектом;  
\- находит статичные интервалы, нерабочие сайты и несоответствия;  
\- помечает признаки поиска работы, сторонних проектов или развлечений как risk flags;  
\- маскирует чувствительные данные;  
\- формирует отчёт для менеджера;  
\- не обвиняет сотрудника и не принимает санкции автоматически.

Правильный режим:

collect screenshots  
→ redact sensitive data  
→ describe activity  
→ classify  
→ compare with task  
→ detect anomalies  
→ score risk  
→ human review  
→ manager action

Плохой агент:

"Нашёл, кто фармит время и кто ищет работу."

Хороший агент:

"Проанализировал 1 240 скриншотов.  
Нашёл 37 эпизодов с низким соответствием задаче.  
12 эпизодов — статичный экран более 30 минут.  
5 эпизодов — сайты вакансий или нерелевантные страницы.  
Все выводы требуют ручной проверки.  
Чувствительные данные замаскированы.  
Автоматических санкций не применялось."
